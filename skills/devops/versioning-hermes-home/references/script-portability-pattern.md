# Script portability pattern for versioned HERMES_HOME scripts

When versioning HERMES_HOME to git, the scripts themselves must be portable —
a clone should work on any machine, not just the original Docker deployment.
This is the pattern that emerged from a full portability audit (Jun 2026).

## The problem

Scripts in `scripts/` routinely hardcode the absolute HERMES_HOME path
(`/opt/data` in Docker, `~/.hermes` on bare metal). When someone clones the
repo to a different path, every script breaks silently. The same applies to
the `hermes` binary path (`/opt/hermes/.venv/bin/hermes` inside the container).

## The fix: env-var indirection with safe defaults

Every script that references HERMES_HOME must use:

```python
import os
from pathlib import Path

HERMES_HOME = Path(os.environ.get("HERMES_HOME", "/opt/data"))
```

Then build all paths from `HERMES_HOME`:

```python
# ❌ BAD — breaks on any other machine
GOOGLE = Path('/opt/data/skills/productivity/google-workspace/scripts/google_api.py')
STATE_PATH = Path('/opt/data/cron/state/my_seen.json')
ENV_PATH = Path('/opt/data/.env')

# ✅ GOOD — works everywhere
GOOGLE = HERMES_HOME / "skills/productivity/google-workspace/scripts/google_api.py"
STATE_PATH = HERMES_HOME / "cron/state/my_seen.json"
ENV_PATH = HERMES_HOME / ".env"
```

For the `hermes` binary (used by auto-update guards and similar):

```python
HERMES = os.environ.get("HERMES_BIN", "/opt/hermes/.venv/bin/hermes")
```

For shell scripts:

```bash
HERMES_HOME="${HERMES_HOME:-/opt/data}"
cd "$HERMES_HOME"
```

## Common patterns to replace

| Hardcoded pattern | Portable replacement |
|---|---|
| `Path('/opt/data/...')` | `HERMES_HOME / "..."` |
| `Path("/opt/data/cron/jobs.json")` | `HERMES_HOME / "cron/jobs.json"` |
| `'/opt/data/state.db*'` (glob) | `str(HERMES_HOME / 'state.db*')` |
| `glob.glob('/opt/data/*.lock')` | `glob.glob(str(HERMES_HOME / '*.lock'))` |
| `"/opt/hermes/.venv/bin/hermes"` | `os.environ.get("HERMES_BIN", "/opt/hermes/.venv/bin/hermes")` |
| `sys.path.insert(0, "/opt/data/scripts")` | `sys.path.insert(0, str(HERMES_HOME / "scripts"))` |
| `sys.path.insert(0, "/opt/hermes")` | `sys.path.insert(0, os.environ.get("HERMES_SOURCE", "/opt/hermes"))` |
| `GIT = '/usr/bin/git'` | `import shutil; GIT = shutil.which('git') or 'git'` |
| `BRIDGE_DIR=/opt/hermes/scripts/...` (shell) | `BRIDGE_DIR="${HERMES_SOURCE:-/opt/hermes}/scripts/..."` |

## Don't forget `import os` (and `import shutil`)

Scripts that previously used only `from pathlib import Path` and never needed
`os` will crash with `NameError: os is not defined` after adding the env-var
pattern. Always add `import os` alongside the new HERMES_HOME line.

If you also replaced a hardcoded binary path (`/usr/bin/git` →
`shutil.which('git')`), add `import shutil` too.

## Don't forget shebangs in library modules

Files like `usaw_to_lib.py` that are imported by other scripts may not have
a `#!/usr/bin/env python3` shebang — the parent script's shebang covers the
process. But if anyone runs the library module directly (for testing), it
fails on systems where `.py` isn't associated with python3. Add shebangs to
all `.py` files during the audit.

## Bulk-patching approach

When fixing many scripts at once (execute_code or a script):

1. Replace the first hardcoded path with `HERMES_HOME = Path(os.environ.get(...))` + the portable version
2. Replace all remaining `/opt/data/...` references in the same file with `HERMES_HOME / "..."`
3. Add `import os` if the file doesn't already have it
4. Run `py_compile` on every modified file — catches string-quoting errors from automated replacements
5. Grep for any remaining `/opt/data` in runtime code (exclude docstrings/comments)
6. Test with `HERMES_HOME=/tmp/test python script.py` to prove it works with a different base

## Pitfall: broken string replacements

When doing automated `Path('/opt/data/cron/state/X.json')` → `HERMES_HOME / "cron/state/X.json"`,
naive string replacement can produce invalid syntax:

```python
# ❌ Broken — unclosed string literal
STATE_PATH = HERMES_HOME / "cron/state/"followup_seen.json')

# ✅ Correct
STATE_PATH = HERMES_HOME / "cron/state/followup_seen.json"
```

The issue: the replacement of `Path('/opt/data/cron/state/` with `HERMES_HOME / "cron/state/`
leaves a dangling `X.json')` that needs to become `"X.json"`. Always verify with
`py_compile` after bulk patches.

## node_modules exclusion

If the repo contains a Node.js bridge or similar with `node_modules/`, it MUST
be in `.gitignore` — these are 5,000+ files, tens of MB, regenerated by `npm install`.
Only track `package.json`, `package-lock.json`, and source files.

```gitignore
scripts/whatsapp-bridge/node_modules/
```

Add a step to the README's deploy instructions:
```bash
cd scripts/whatsapp-bridge && npm install && cd ../../
```

## .gitattributes for cross-platform

Add `.gitattributes` to normalize line endings so clones on macOS/Linux/Windows
don't produce CRLF/LF churn in diffs:

```gitattributes
* text=auto eol=lf
*.sh text eol=lf
*.py text eol=lf
*.yaml text eol=lf
*.json text eol=lf
```

And add `!/.gitattributes` to the `.gitignore` allow-list.

## Verification checklist after portability overhaul

- [ ] `grep -rn '/opt/data' scripts/ --include=*.py` shows only docstrings/comments
- [ ] `grep -rn '/opt/hermes' scripts/ --include=*.py` shows only defaults inside `os.environ.get(...)`
- [ ] `grep -rn '/usr/bin/' scripts/ --include=*.py` shows only `shutil.which(...)` results
- [ ] All modified scripts pass `python3 -m py_compile`
- [ ] All modified shell scripts pass `bash -n`
- [ ] All `.py` files have `#!/usr/bin/env python3` shebang
- [ ] `HERMES_HOME=/tmp/test python3 scripts/some_script.py` runs without crash
- [ ] Remote GitHub tree has zero `node_modules/` entries
- [ ] `.gitattributes` exists and is tracked
- [ ] `README.md` documents the env vars and deploy steps

## Complete env-var convention

| Variable | Default | Used For |
|----------|---------|----------|
| `HERMES_HOME` | `/opt/data` | All data paths (config, scripts, cron, state, wiki, skills) |
| `HERMES_SOURCE` | `/opt/hermes` | Hermes source code root (for Python imports, bridge paths) |
| `HERMES_BIN` | `/opt/hermes/.venv/bin/hermes` | Hermes CLI binary path |
| `HERMES_TZ` | `America/Los_Angeles` | Timezone for all cron output (cron_style.py LOCAL_TZ) |
| `USAW_STATE_DIR` | `$HERMES_HOME/cron_state/usaw_to` | USAW watcher state (job-specific) |
| `WIKI_PATH` | `$HERMES_HOME/wiki` | Wiki knowledge base path |

The convention: **every absolute path** in a script should resolve through
one of these env vars with a Docker-default fallback. No bare literals.

## Timezone portability (cron_style.py)

The shared `cron_style.py` module hardcodes `LOCAL_TZ = ZoneInfo("America/Los_Angeles")`.
If the deployment moves to a different timezone (user travels, new machine in
another TZ), all cron output times are wrong. Fix:

```python
import os
from zoneinfo import ZoneInfo

LOCAL_TZ = ZoneInfo(os.environ.get("HERMES_TZ", "America/Los_Angeles"))
```

Also check for standalone `ZoneInfo("America/Los_Angeles")` in individual
scripts (calendar_travel_alerts.py, eod_wrap_precheck.py, etc.) — each should
respect the same env var, or import LOCAL_TZ from cron_style.

## config.yaml Docker-specific URLs

The `config.yaml` provider section may use `host.docker.internal:11434` —
this is Docker's bridge to the host machine and only works inside containers.
For non-Docker deployments, change to `localhost:11434`. Document this in the
README's Docker section so a fresh clone knows to adjust.

## cron_review_prompt.md is a frozen reference

The shared `cron_review_prompt.md` (house-style guide for LLM cron jobs) is
**frozen at job creation time** — editing the file does NOT update existing
cron jobs. Add a comment at the top of the file noting this, so future agents
don't edit it expecting live jobs to pick up changes:

```html
<!-- REFERENCE TEMPLATE — Hermes cron jobs freeze their full prompt at creation time.
     Editing this file does NOT update existing jobs. To change a job's prompt, use:
     cronjob action=update job_id=<id> prompt="..." -->
```

## Deep audit: holistic script review

Beyond path portability, a full audit should evaluate scripts as a system:

1. **Undefined variables** — `grep` for common bugs: variables referenced before
   assignment (e.g. `wiki_path` used when the local is `root`). Run `py_compile`
   to catch syntax errors, but also read each script's logic flow.
2. **Account aliases** — Scripts may hardcode Google account names (`'personal'`,
   `'nonprofit'`). These are deployment-specific. The gold standard is reading
   from `google/accounts.json`, but accepting hardcoded fallbacks for personal
   config is reasonable.
3. **Shared helper duplication** — The `run()` + `UV_GOOGLE_DEPS` pattern is
   deliberately copy-pasted (script-first "no cross-imports" rule). A test
   harness (`test_cron_prechecks.py`) catches structural drift. This is correct
   architecture, not a code smell.
4. **One-off scripts** — Event-specific scripts (e.g. `ncw_alerts.py` with a
   hardcoded meet schedule, `remind_morocco_brazil.sh` for a specific match)
   should be moved to `archived/` after the event passes.
5. **qmd binary path** — Use `shutil.which("qmd")` with a fallback to the
   npm-global path, not a bare hardcoded path.
