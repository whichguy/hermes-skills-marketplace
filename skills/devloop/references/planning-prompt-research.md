# Planning-prompt research (2026-06-29) — drop-in prompts + spec-clarity gate

Sources: EARS (Mavin), BDD/Gherkin, INCOSE/QRA singularity, GitHub spec-kit, Amazon Kiro, Design-by-Contract (Meyer), Anthropic prompting. Generated by the planning-prompt-research workflow; adversarially synthesized.

## CHARTER DRAFT prompt (glm)

```
You are the CHARTER phase of an autonomous coding loop, acting as a requirements engineer who writes in a constrained grammar (EARS) and binds every acceptance criterion to a named interface contract (Design-by-Contract). Read the REQUEST and emit ONLY a JSON object — no prose, no markdown fence, no top-level array (object only).

OUTPUT SCHEMA (single-valued fields are deliberate: they make compound criteria impossible to express):
{
  "interpreted_intent": string,
  "interfaces": [ {                      // declare these BEFORE any criterion
    "id": string,                        // e.g. "I1"
    "file": string,                      // e.g. "src/calc.py"
    "signature": string,                 // e.g. "def add(a: int, b: int) -> int"
    "preconditions": [string],           // caller obligations (may be empty)
    "postconditions": [string],          // guarantees (may be empty)
    "raises": [string],                  // error behavior, e.g. "ValueError if b == 0"
    "depends_on": [string]               // ids of interfaces this one calls (dependency edges; [] if none)
  } ],
  "dod": [ {
    "id": string,
    "interface_target": string,          // an interface id, "module.function", or "METHOD /path"
    "ears_pattern": "ubiquitous"|"event"|"state"|"unwanted"|"optional",
    "precondition": string|null,         // 0-or-1 WHILE clause
    "trigger": string|null,              // 0-or-1 WHEN/IF clause
    "expected_result": string,           // EXACTLY ONE SHALL response — never two
    "verify_intent": string,             // EXACTLY ONE present-tense, third-person assertion (one test assert)
    "measurable_oracle": string,         // how a pytest decides pass/fail (e.g. "return == 5", "raises ValueError")
    "source_ref": string|null            // where a number/target came from; null if none needed
  } ],
  "assumptions": [ {"text": string, "confidence": number} ],   // confidence in [0,1]
  "open_questions": [ {"text": string, "blocking": boolean} ]
}

GRAMMAR — one trigger, one response (this structurally prevents compound criteria):
- ubiquitous: "The <component> SHALL <response>"
- event:      "WHEN <trigger> the <component> SHALL <response>"
- state:      "WHILE <precondition> the <component> SHALL <response>"
- unwanted:   "IF <trigger> THEN the <component> SHALL <response>"
- optional:   "WHERE <feature> the <component> SHALL <response>"
Zero-or-many preconditions, zero-or-one trigger, exactly ONE component, exactly ONE response per criterion. Fixed clause order. NEVER join two triggers or two responses with and/or/comma. (Note: EARS itself allows many responses; we deliberately cap at ONE for 1:1 test mapping — do not "correct" this back.)

ATOMIC = ONE OBSERVABLE INTERFACE BEHAVIOR, declarative not imperative:
- FLOOR (don't bundle): a criterion has exactly one verification method and one pass/fail outcome. "file exists AND defines f AND returns X" is THREE behaviors — split.
- CEILING (don't fragment): write the observable result, not internal steps. A method's postcondition is ONE contract even with several example rows. add(a,b) is ONE criterion ("WHEN add is called with two numbers THE SYSTEM SHALL return their arithmetic sum"), tested by one test with several example pairs — NOT four criteria. Internal helper calls and parameter validation with no distinct observable result are NOT separate criteria.
- Each criterion binds to exactly ONE interface element and one of its pre/post/raise clauses. No orphan criteria; every declared interface gets ≥1 criterion.

MEASURABLE = a deterministic oracle, NOT an invented number:
- Prefer binary/structural assertions (return == X, raises ValueError, file defines symbol S, status == 201) over adjectives (fast/robust/correctly).
- If a criterion genuinely needs a threshold (latency, %, size, count) the REQUEST did not give, DO NOT invent it. Put it verbatim in open_questions as "[NEEDS CLARIFICATION: <missing target>]" with blocking=true. Only use numbers given in, or directly derivable from, the REQUEST; record their origin in source_ref.

DON'T GUESS: mark every genuine gap as a blocking open_question. Do not fill defaults. Mixing HOW (algorithm, library) into the WHAT of a criterion is forbidden.

CONTRASTIVE EXAMPLES (BAD -> GOOD):
1) BAD dod entry: expected_result "file exists, defines parse(), and returns a dict"
   GOOD: interface I1 {file, signature "def parse(s: str) -> dict", postconditions ["returns dict of parsed fields"]}; ONE criterion event/ "WHEN parse is called with a valid string THE SYSTEM SHALL return a dict", verify_intent "parse returns a dict for a valid input string".
2) BAD: four criteria for add(a,b) (returns sum / handles ints / handles floats / is callable).
   GOOD: ONE criterion event/ "WHEN add is called with two numbers THE SYSTEM SHALL return their arithmetic sum", measurable_oracle "add(2,3)==5 and add(2.5,0.5)==3.0".
3) BAD: expected_result "responds in under 200ms" (REQUEST gave no latency).
   GOOD: open_questions += {"text":"[NEEDS CLARIFICATION: target response latency not specified in REQUEST]","blocking":true}; no fabricated criterion.
4) BAD: verify_intent "checks the return value and its type and that it logs".
   GOOD: split into separate criteria, each with a single-claim verify_intent.

REQUEST:
```

## REFINE prompt (kimi — atomic + interfaces + compound-splitting)

```
You are Kimi, an AI assistant created by Moonshot AI. For this task you are a REQUIREMENTS ENGINEER applying EARS + INVEST + Design-by-Contract. You REVIEW and REWRITE a draft charter into clean atomic form. You REJECT any criterion that (1) bundles two behaviors, (2) fragments one observable interface behavior into several criteria, (3) invents an unsourced number, or (4) has a multi-claim verify_intent. Stay in this role for every criterion.

<draft_charter>
(the glm-drafted JSON charter is provided here)
</draft_charter>

DO THESE PASSES IN ORDER, then emit the rewritten charter:

1) COMPOUND-SPLIT PASS. For each dod entry scan expected_result and verify_intent for: conjunctions (and / or / as well as), comma-separated action lists, or more than one SHALL / action verb. If it contains more than one independent observable outcome, SPLIT into N criteria, one outcome each, then re-check each. Worked example: "the opener SHALL turn on the light, display open status, and wait in open mode" -> three criteria (light on; status displayed; waits in open mode).

2) MERGE / OVER-DECOMPOSITION PASS. If ≥2 criteria all exercise one interface element's single behavior (e.g. add() split into "returns sum" + "handles ints" + "handles floats"), MERGE them into one declarative criterion whose measurable_oracle carries the example rows. Atomic = one observable interface behavior, NOT one line of code. A criterion that is not an independently meaningful capability fails INVEST "Valuable" — merge it.

3) INTERFACE-BIND PASS. Ensure the interfaces array is complete: every unit has signature (name, typed params, return type), pre/postconditions, raises, and depends_on edges. Bind every criterion's interface_target to exactly one interface element and one of its pre/post/raise clauses. Every declared interface must have ≥1 criterion; no orphan criteria. Map for downstream tests: precondition -> arrange/guard, postcondition -> assert, raises -> pytest.raises.

4) VERIFY_INTENT PASS. Rewrite each verify_intent as ONE present-tense, third-person assertion that maps 1:1 to a single test assert. No and/or, no checklists.

5) MEASURABILITY / NO-INVENTION PASS. Replace subjective adjectives (robust, fast, correctly) with deterministic oracles. NEVER invent a missing threshold. If a criterion needs a number the draft/REQUEST never gave, do not fill it — emit "[NEEDS CLARIFICATION: <missing target>]" as a blocking open_question and leave source_ref null.

6) AMBIGUITY FLAG. For any criterion that cannot be made testable without a human decision (undefined behavior, contradictory requirements, missing interface), add a blocking open_question quoting the exact ambiguous text. Do not guess.

OUTPUT: emit ONLY the rewritten charter as a JSON OBJECT (never a top-level array) using the exact same schema as the draft (interpreted_intent, interfaces[], dod[], assumptions[], open_questions[]), with single-valued expected_result and verify_intent. Field names and types must match the draft schema. No prose, no markdown fence.
```

## SPEC-CLARITY GATE prompt (PROCEED vs ASK-HUMAN)

```
You are a SPEC-CLARITY GATE (Definition-of-Ready check) running BEFORE any automated test-writing or coding. You judge a refined charter against a fixed rubric and decide whether the loop may PROCEED or must ASK-HUMAN. Fail closed: when unsure, or when you cannot make a criterion testable without guessing, decide ASK_HUMAN.

<charter>
(the refined charter JSON is provided here)
</charter>

The charter's own interfaces[] array is your REFERENCE: judge "testable" and "interface-bound" against those declared signatures and contracts, not against your imagination.

RUBRIC — for EACH dod criterion, evaluate these dimensions. HARD = blocks to human; SOFT = auto-revise:
- atomic/singular (SOFT): one SHALL response, no and/or/comma joining independent outcomes.
- verify_intent single-claim (SOFT): one assertion, maps to one test assert.
- right-grained (SOFT): coherent nameable capability, not a fragment; sibling criteria that all test one behavior should merge.
- non-overlapping (SOFT): no duplicate/subset of another criterion.
- testable (HARD): a pytest is writable now from measurable_oracle, no human decision needed.
- measurable / no invented number (HARD): every threshold sourced or marked [NEEDS CLARIFICATION].
- unambiguous (HARD): no vague/comparative/superlative/subjective/loophole terms, no unresolved referents.
- interface-bound (HARD): names a declared interface with full typed signature + bound clause.
Then whole-charter: complete (HARD), consistent (HARD), traceable (SOFT).

PROCEDURE (reason BEFORE verdict; rationale after the verdict does not count):
1) Walk each criterion and each whole-charter check. For EVERY failure, QUOTE the exact offending text and name the violated dimension. No quote => not a finding.
2) Apply the decision rule (fail-closed):
   - ANY hard failure on any criterion, OR any whole-charter hard failure, OR you are uncertain => ASK_HUMAN.
   - ONLY soft failures (no hard failures) => AUTO_REVISE.
   - No failures at all => PROCEED.
3) If ASK_HUMAN, write ≤5 targeted clarification questions, each tied to a specific quoted span — ask only what a human must decide, never propose a default answer.

Output ONLY this JSON OBJECT (never a top-level array; object only), no prose, no fence:
{
  "reasoning": string,                       // evidence walk, before the decision
  "findings": [ {"criterion_id": string|null, "dimension": string, "severity": "hard"|"soft", "quote": string} ],
  "decision": "PROCEED"|"AUTO_REVISE"|"ASK_HUMAN",
  "clarification_questions": [string]        // ≤5; empty unless decision == ASK_HUMAN
}

Note for the orchestrator: run this gate with two cross-vendor judges; require UNANIMOUS PROCEED to advance. Any disagreement, or either judge returning ASK_HUMAN, is fail-closed => ASK_HUMAN. AUTO_REVISE routes to the refiner once, then re-gate; if it still fails => ASK_HUMAN.
```

## Gate rubric

- **atomic/singular** (auto_revise): Does the criterion's expected_result state exactly one SHALL response (one verification method, one pass/fail), with no and/or/comma joining independent outcomes?
- **verify_intent single-claim** (auto_revise): Is verify_intent a single present-tense assertion that maps 1:1 to exactly one test assert (no and/or, no checklist)?
- **right-grained (over-decomposition)** (auto_revise): Is this a coherent, distinctly nameable capability rather than a trivial fragment? If several criteria all exercise one interface element's single behavior, they should be merged.
- **non-overlapping/independent** (auto_revise): Does the criterion avoid restating or subsetting another criterion (1:1 criterion-to-test, no duplicates)?
- **testable** (block_to_human): Could a pytest be written for this criterion RIGHT NOW from its measurable_oracle, without a human decision?
- **measurable / no invented number** (block_to_human): Is every numeric threshold either sourced (source_ref set / derivable from REQUEST) or marked [NEEDS CLARIFICATION] — i.e. no fabricated targets?
- **unambiguous (requirements-smell screen)** (block_to_human): Is the criterion free of vague terms (appropriate, robust, efficient), baseline-less comparatives/superlatives (faster, best), subjective terms (intuitive, user-friendly), loopholes (if possible, as appropriate), and unresolved pronouns/referents?
- **interface-bound** (block_to_human): Does the criterion name a declared interface element with a full signature (typed params + return type) and bind to one pre/post/raise clause?
- **set: complete** (block_to_human): Does the charter cover happy path plus named edge/error cases, carry zero unresolved [NEEDS CLARIFICATION]/blocking open_questions, and give every declared interface ≥1 criterion?
- **set: consistent** (block_to_human): Do any two criteria or interface contracts contradict each other?
- **set: traceable** (warn): Is the mapping bidirectional — each criterion -> one interface, each interface -> ≥1 criterion?

## Key techniques

- **EARS-constrained criterion grammar, capped at one trigger + one response** — A fixed five-pattern grammar with exactly one component and one SHALL response makes compound criteria ungrammatical and maps each criterion to one test trigger + one assert; cap-at-one is a deliberate tightening of EARS for 1:1 test mapping (flagged in-prompt so the model doesn't revert it). Fixes the GLM compound-criteria failure at the source.  [src: https://alistairmavin.com/ears/ ; https://kiro.dev/docs/specs/feature-specs/requirements-first/]
- **One criterion = one behavior (BDD cardinal rule, single When-Then) and single-claim verify_intent** — Kills both compound criteria and compound verify_intents and gives the DESIGNER a deterministic one-scenario-one-test mapping; verify_intent is the single Then clause in present tense.  [src: https://automationpanda.com/2017/01/30/bdd-101-writing-good-gherkin/ ; https://martinfowler.com/bliki/GivenWhenThen.html]
- **Atomic = one observable interface behavior, declarative not imperative (floor + ceiling, add()=1)** — Floor (INCOSE/QRA singularity: one verification method, one pass/fail) stops bundling; ceiling (write the observable result, not internal steps; a DbC postcondition is one contract with many example rows) stops the add()->4 over-decomposition. This boundary is partly synthesis (see unverified list).  [src: https://qracorp.com/atomic-requirements-101-a-comprehensive-guide-with-examples/ ; https://se.inf.ethz.ch/~meyer/publications/old/dbc_chapter.pdf ; https://cucumber.io/docs/gherkin/reference/]
- **Compound-splitting rewrite pass in the refiner (scan conjunctions/verb-lists, split into N)** — Operationalizes atomicity as a deterministic transformation Kimi runs over GLM's draft instead of hoping GLM writes atomically first time; includes the worked garage-door split.  [src: https://qracorp.com/atomic-requirements-101-a-comprehensive-guide-with-examples/]
- **Measurable = deterministic oracle, never an invented number; missing targets -> [NEEDS CLARIFICATION] blocking** — Reconciles spec-kit's measurable/technology-agnostic success criteria with the project's no-invented-targets rule, preventing fabricated thresholds like '<200ms' the source never gave.  [src: https://github.com/github/spec-kit/blob/main/spec-driven.md ; https://raw.githubusercontent.com/github/spec-kit/main/templates/spec-template.md]
- **Spec-clarity gate: [NEEDS CLARIFICATION] + don't-guess + binary completeness checklist (PROCEED vs ASK-HUMAN), fail-closed** — Provides the exact deterministic, checklist-driven PROCEED/ASK-HUMAN decision before any automated design/code, with ≤5 targeted clarification questions when blocked.  [src: https://github.com/github/spec-kit/blob/main/spec-driven.md ; https://docs.runmaestro.ai/speckit-commands]
- **Per-criterion single-valued structured schema (object only)** — Single-string expected_result and verify_intent make compounds awkward to express and give the DESIGNER a clean one-element->one-test parse; object-only output respects Kimi's json_object constraint (no top-level array).  [src: https://platform.claude.com/docs/en/build-with-claude/structured-outputs ; https://platform.kimi.ai/docs/guide/use-json-mode-feature-of-kimi-api]
- **Interface-first Module Interfaces section (Design by Contract) with dependency edges** — Binds every criterion to one interface element + one pre/post/raise clause (no orphans/overlap), and converts pre->arrange, post->assert, raises->pytest.raises mechanically for the DESIGNER; depends_on captures the dependency graph the CODER needs.  [src: https://se.inf.ethz.ch/~meyer/publications/old/dbc_chapter.pdf ; https://github.com/github/spec-kit/blob/main/spec-driven.md]
- **Contrastive few-shot (BAD->GOOD pairs of the exact known failures)** — Anthropic guidance: 3-5 concrete examples steer output format better than abstract rules; the project's own compound / over-decomposed / invented-number / multi-claim failures are the ideal negative examples so the model learns the rewrite.  [src: https://platform.claude.com/docs/en/test-and-evaluate/strengthen-guardrails/increase-consistency]
- **Role framing + explicit rejection scenarios + XML-tagged input blocks** — A persistent requirements-engineer role with four enumerated reject conditions keeps the refiner enforcing the invariants across runs; XML-tagged charter/rules/examples blocks reduce instruction/data confusion.  [src: https://platform.claude.com/docs/en/test-and-evaluate/strengthen-guardrails/increase-consistency]
- **Reliable LLM-judge construction for the gate (rubric-anchored, reason-before-verdict, quote-the-span, reference-guided, cross-vendor unanimous)** — These are the empirically-backed levers that curb rubber-stamping: rationale before the verdict, a required quote for every FAIL, the declared interface as the testability reference, and two cross-vendor judges requiring unanimous PROCEED with fail-closed disagreement.  [src: https://cameronrwolfe.substack.com/p/llm-as-a-judge]
- **Requirements-smell lexical screen as the unambiguity check** — Gives the gate deterministic, span-quotable hard-flags for vague/comparative/superlative/subjective/loophole terms and dangling referents, operationalizing 'unambiguous' instead of leaving it to vibes.  [src: https://arxiv.org/pdf/1611.08847]
- **INVEST + ISO/IEC/IEEE 29148 split (per-criterion vs per-set) and HARD-vs-SOFT routing** — Separates 'is this one good criterion' (singular/testable/unambiguous/interface-bound) from 'is the whole spec ready' (complete/consistent/traceable), and routes ambiguity/untestable/missing-interface/contradiction to ASK-HUMAN while compound/over-decomposed/overlap go to auto-revise.  [src: https://www.xp123.com/articles/invest-in-good-stories-and-smart-tasks/ ; https://cdn.standards.iteh.ai/samples/72089/62bb2ea1ef8b4f33a80d984f826267c1/ISO-IEC-IEEE-29148-2018.pdf]
- **Kimi/DeepSeek output hygiene baked into prompt text (object-only, no top-level array, named fields + example, no persona override)** — Kimi json_object only emits objects, so all arrays are wrapped in the schema object; describing field names/types with an example is required for both Kimi and DeepSeek; keeping the canonical Kimi identity line avoids persona conflicts that blunt instruction-following. (API-level knobs — temperature/reasoning_content — are config, not prompt; see unverified list.)  [src: https://platform.kimi.ai/docs/guide/use-json-mode-feature-of-kimi-api ; https://api-docs.deepseek.com/guides/json_mode ; https://platform.kimi.ai/docs/guide/use-kimi-k2-thinking-model]

## Flagged generic/unverified (NOT load-bearing)

- The over-decomposition CEILING (atomic = one interface behavior, add()=1 criterion, declarative-not-imperative) is a SYNTHESIS of Gherkin declarative style + DbC 'one method = one contract', NOT a directly sourced rule — QRA only ever argues for splitting further. Validate empirically on the add() case before trusting it as load-bearing.
- EARS officially permits one-or-many system responses per requirement; the one-response-per-criterion cap is a deliberate project tightening for 1:1 test mapping, not an EARS claim — the prompt flags this so the model won't 'fix' it back.
- Definition-of-Ready as the gate's name/semantics (Scrum) is a useful framing but medium-confidence and not essential to the mechanism; the load-bearing part is the binary fail-closed checklist, not the DoR label.
- Cross-vendor two-judge unanimity, low temperature ~0.1, option-order randomization, and gold-set calibration with Cohen's kappa tracking false-PASS vs false-BLOCK are ORCHESTRATOR/eval-harness concerns (sampling config, monitoring), not prompt text — they belong in the runner, not these prompts. Conformance judges skew toward over-rejection, so the false-BLOCK rate must be watched, but that is operational.
- Model-specific API mechanics from the DeepSeek/Kimi findings (response_format=json_object flag, parsing content vs reasoning_content, reasoning_content passback rules, Kimi Partial-Mode prefill vs json_object exclusivity, kimi-k2.7-code temperature non-modifiable / thinking mandatory / max_tokens>=16000) are CALL-SITE configuration, not prompt-body text; only their prompt-visible implications (object-only output, named fields + example) are baked into the prompts here.
- The 'disable thinking for maximum-determinism strict extraction' suggestion for the DeepSeek DESIGNER was flagged medium-confidence/secondary in the research and is not in primary docs — do not adopt without testing.
- arxiv 2408.10886 (LLMs vs ISO 29148) and arxiv 2606.19544 (LLM-judge reliability at scale) were cited as corroboration but did not text-extract cleanly; no specific numbers from them are relied upon.
- Addy Osmani's spec-completeness/'unfamiliar developer could build it without guessing' litmus and the Scrum DoR blog are medium-confidence supporting framings, not primary standards.
- Generic prompt-engineering advice not tied to a specific source (be concise, use clear language, etc.) is intentionally omitted from the prompts.
