# v3.1 Implementation Bugs — Integration Session (2026-06-29)

**Context:** Kimi K2.7 Code implemented `sdlc_parallel.py` (515 lines) + state machine integration from the v3.1 design doc. The implementation compiled but 7 integration tests failed. All 8 bugs below were found and fixed during the integration session. Final result: 17/17 tests passing (7 integration + 10 E2E).

## Bug 1: `_git_or_raise` double-prefix (`git git add`)

**Symptom:** `git git add -f -A` in command output, git commands failing silently.

**Root cause:** `_git_or_raise()` prepends `"git"` to the command list (`full_cmd = ["git"] + cmd`), but call sites passed `["git", "add", "-f", "-A"]` — producing `["git", "git", "add", "-f", "-A"]`.

**Fix:** Remove `"git"` from all `_git_or_raise` call args. The helper already adds it.

**Detection:** `grep -n '_git_or_raise(\["git"' sdlc_parallel.py` — any match is a bug.

**Lesson:** When a helper wraps a command prefix, audit ALL call sites for the prefix. One `grep` catches every instance.

## Bug 2: Shallow copy not stored back into tasks list

**Symptom:** `cwd=''` in dispatch — child processes ran in wrong directory, files written to wrong location.

**Root cause:** 
```python
for i, task in enumerate(tasks):
    task = dict(task)        # shallow copy → local variable
    task["cwd"] = wt_path    # set on local copy
    # BUG: local `task` never stored back into `tasks[i]`
```
Later iteration `for task in tasks` gets the original dict without `cwd`.

**Fix:** `tasks[i] = task` after setting `cwd` on the copy.

**Lesson:** When you mutate a loop variable that's a copy of a list element, you MUST store it back. Python's `for` loop rebinds the variable each iteration — mutations to the variable don't affect the list.

## Bug 3: Branch deletion order (worktree before branch)

**Symptom:** `git branch -d <child-branch>` fails with "cannot delete branch checked out at <worktree>".

**Root cause:** `git branch -d` refuses to delete a branch that's checked out in any worktree. The code called `git branch -d` BEFORE `remove_worktree()`.

**Fix:** Swap the order — `remove_worktree(wt_path)` first, then `git branch -d child_branch`.

**Lesson:** Git won't delete a branch while any worktree has it checked out. Always remove the worktree first.

## Bug 4: Merge conflict test needs pre-existing file

**Symptom:** Both children write `shared.py` with different content, but merge succeeds with no conflict. Test expects conflict.

**Root cause:** When both children add a NEW file (not in the base commit), git's 3-way merge sees no common ancestor for that file. Both "adds" are independent — git auto-resolves by taking one version. No conflict is raised.

**Fix:** Pre-create `shared.py` in the base commit with initial content. Both children then MODIFY the same file — git sees a common ancestor and raises a conflict when both sides diverge.

**Lesson:** To test merge conflicts, files must exist in the merge base. Fresh adds from both sides don't conflict.

## Bug 5: Module-level constant read at import time

**Symptom:** `test_01` changes `os.environ["SDLC_WORKTREE_ROOT"]`, but `test_02` still uses the old value. Tests pass in isolation but fail when run together.

**Root cause:** `WORKTREE_ROOT = os.environ.get("SDLC_WORKTREE_ROOT", ...)` is evaluated once at module import time. Tests that change the env var after import don't affect the already-read constant.

**Fix:** Tests must set `os.environ["SDLC_WORKTREE_ROOT"]` BEFORE importing the module, or the module must re-read the env var at call time (not import time).

**Lesson:** Module-level constants from env vars are frozen at import. Tests that change env vars must do so before import, or the code must use a function that reads the env var lazily.

## Bug 6: Subagent reformats unrelated files

**Symptom:** `git diff --cached` showed 342 lines changed in `sdlc_worktree.py` — docstring rewrites, spacing fixes, comment rephrasing. Kimi was only supposed to add new files and modify `sdlc_state.py`.

**Root cause:** Subagents with file write access can "helpfully" clean up files they read, even when not asked. Kimi read `sdlc_worktree.py` for context and decided to improve its docstrings.

**Fix:** `git checkout HEAD -- sdlc_worktree.py` to revert all unintended changes before committing.

**Lesson:** After ANY subagent implementation session, run `git diff --stat` and revert files the subagent shouldn't have touched. Subagents are autonomous — they will "improve" things you didn't ask for.

## Bug 7: Stray test artifacts in git index

**Symptom:** `module_.py`, `module_ok.py`, `module_sym.py`, `module_unknown.py`, `module_x.py`, `shared.py` appeared in `git diff --cached` even after being deleted from disk.

**Root cause:** Mock functions in the test suite wrote these files to the scripts directory. They were committed in a previous commit, then deleted from disk by cleanup. But `git add -A` re-stages deletions from the index — and if the files were in a prior commit, they show as "new file" in the diff against that commit.

**Fix:** `git rm --cached` each file, then commit the removal.

**Lesson:** Test mocks that write files to the project directory leave artifacts. Always check `git status` after test runs and clean up. Consider writing test artifacts to `/tmp/` instead.

## Bug 8: `git rm --cached` then `git add -A` re-stages deleted files

**Symptom:** After `git rm --cached module_*.py`, running `git add -A` brought them back.

**Root cause:** `git rm --cached` removes from the index but doesn't commit. `git add -A` sees the files still in the working tree (or in a prior commit) and re-adds them. The removal must be committed before `git add -A` is run again.

**Fix:** Either commit the `git rm --cached` before running `git add -A`, or use `git reset HEAD -- <files>` to unstage them and then ensure they're deleted from disk before `git add -A`.

**Lesson:** `git rm --cached` is not persistent until committed. Any subsequent `git add -A` will re-stage the files if they exist on disk or in a prior commit.
