# v3.1 Concurrent Dispatch Design — Option A + API Communication Pathway

> Generated: 2026-06-29
> Status: APPROVED WITH CHANGES — P0/P1 fixes applied per advisor review (2026-06-29)
> Scope: Plan out Option A (ThreadPoolExecutor concurrent dispatch) with a forward-looking
> design for API-based child↔orchestrator communication (child can ask questions / converse)

---

## 1. Problem Statement

The SDLC v6 orchestrator is strictly sequential: one `dispatch_single()` call blocks the
entire while-loop. When the PLAN phase identifies 3 independent modules, the orchestrator
dispatches 3 sequential IMPLEMENT calls (~100s each = 300s total). With concurrent dispatch,
this drops to ~max(100s, 100s, 100s) = ~100s — a 3x speedup.

Additionally, the current architecture has no mechanism for a child process to ask the
orchestrator a question mid-flight. The child is a headless `hermes chat -q --yolo` subprocess
that returns one text blob at exit. If the coder encounters an ambiguity in PROJECT.md, it
can't ask for clarification — it either guesses wrong (producing broken code) or produces
nothing (triggering a debug cycle that wastes another dispatch).

This document designs:
- **Phase 1 (Option A):** Concurrent dispatch via ThreadPoolExecutor (no child→orchestrator messaging)
- **Phase 2 (API pathway):** How to integrate bidirectional communication so children can
  ask questions or converse with the orchestrator

---

## 2. Phase 1 — Concurrent Dispatch (Option A)

### 2.1 Existing Foundation

We already have a working concurrent dispatch pattern: `dispatch_comparison()` in
`model_utils.py:828`. It uses `ThreadPoolExecutor` to dispatch the same prompt to multiple
models in parallel. The key constraints it identified:

1. **`thinking` level is a global config mutation** — `set_reasoning_effort()` calls
   `hermes config set agent.reasoning_effort <level>`, which is process-wide. Parallel
   calls with different thinking levels race. The current workaround: serialize when
   thinking is set, parallelize when it's None.

2. **`subprocess.run()` is thread-safe** — Python's subprocess module is safe to call
   from multiple threads. Each thread blocks on its own subprocess, but the GIL releases
   during `subprocess.run()` I/O waits, so threads don't block each other.

3. **`as_completed()` enables streaming results** — we can process results as they arrive
   rather than waiting for all to finish.

### 2.2 New Function: `dispatch_parallel()`

```python
def dispatch_parallel(
    tasks: list[dict],      # list of dispatch_single kwargs
    max_workers: int = 3,
    timeout: int = 3600,
    progress_callback: Optional[Callable] = None,
) -> dict[str, dict]:
    """Dispatch multiple independent tasks concurrently via ThreadPoolExecutor.

    Each task is a dict of kwargs for dispatch_single(). Tasks run in parallel
    threads, each blocking on its own subprocess.run(). Results are collected
    as they complete via as_completed().

    Constraints:
    - All tasks MUST use thinking=None (or the same thinking level — no per-task
      thinking levels due to global config race). If any task has thinking set,
      all tasks run sequentially (fallback to current behavior).
    - Each task should have a unique 'task_id' key for result identification.
    - Tasks writing to the same worktree MUST be isolated (see §2.3).
    - All tasks get alias=None and output_file=None forced to prevent session
      registry race (last-writer-wins on save_session) and output file collision.
      (P1 fix per advisor review — Seat 1 found this)

    Returns:
        Dict mapping task_id → dispatch_single result dict.
        Each result dict has an added 'task_id' key.
    """
    import concurrent.futures

    # Safety check: if any task has thinking set, serialize all
    has_thinking = any(t.get("thinking") for t in tasks)
    if has_thinking:
        # Fallback: sequential dispatch (current behavior)
        results = {}
        for task in tasks:
            tid = task.pop("task_id", f"task_{len(results)}")
            r = dispatch_single(**task)
            r["task_id"] = tid
            results[tid] = r
            if progress_callback:
                _safe_callback(progress_callback, {
                    "event": "task_complete", "task_id": tid,
                    "success": r.get("content") is not None,
                })
        return results

    # Parallel path: all thinking=None → no config mutation → safe
    results = {}
    with concurrent.futures.ThreadPoolExecutor(max_workers=max_workers) as pool:
        futures = {}
        for task in tasks:
            tid = task.pop("task_id", f"task_{len(futures)}")
            future = pool.submit(dispatch_single, **task)
            futures[future] = tid

        for future in concurrent.futures.as_completed(futures):
            tid = futures[future]
            try:
                r = future.result(timeout=timeout)
            except concurrent.futures.TimeoutExpired:
                r = {"content": None, "error": "parallel dispatch timeout",
                      "elapsed": timeout, "session_id": None}
            except Exception as e:
                r = {"content": None, "error": str(e),
                      "elapsed": 0, "session_id": None}
            r["task_id"] = tid
            results[tid] = r
            if progress_callback:
                _safe_callback(progress_callback, {
                    "event": "task_complete", "task_id": tid,
                    "success": r.get("content") is not None,
                    "elapsed": r.get("elapsed", 0),
                })
    return results
```

### 2.3 Worktree Isolation — The Hard Problem

The biggest risk with concurrent dispatch is **file race conditions**. If two coder
children write to the same worktree simultaneously:

- Both call `git add -A` → one overwrites the other's staging
- Both write `test_solution.py` → last writer wins, first writer's tests lost
- Both modify `solution.py` → import conflicts, merge conflicts
- Both run `ruff --fix` → one's fix undoes the other's fix

**Solution: Per-child git branches, merge after all complete.**

```python
def dispatch_parallel_isolated(

    the orchestrator merges branches sequentially into main.

    1. For each task: git checkout -b task-{id}
    2. Dispatch all tasks in parallel (each writes to its branch's working tree)
    3. Wait for all to complete (barrier)
    4. git checkout main
    5. For each successful task: git merge task-{id} (sequential, conflict-safe)
    6. Clean up: git branch -D task-{id}
    """
```

#### 2.3.1 Correct Approach: `git worktree add` (not branches)

`git checkout -b` changes the working tree for the entire process — can't parallelize
that. The correct primitive is `git worktree add`, which creates a separate working
directory for each branch:

```python
def dispatch_parallel_isolated(
    tasks: list[dict],
    worktree: str,
    base_ref: str = None,
    max_workers: int = 3,
    timeout: int = 3600,
    progress_callback: Optional[Callable] = None,
) -> dict[str, dict]:
    """Concurrent dispatch with per-child git worktree isolation.

    Each task gets its own git worktree (separate directory, shared .git).
    After all tasks complete, the orchestrator merges results back to the
    parent worktree's branch.

    Uses the PATCH FILE WORKFLOW (DeepSeek corner-case analysis, 2026-06-29):
    1. Capture orchestrator's uncommitted state via git diff + untracked file copy
    2. Create child worktree from HEAD, apply patch + copy untracked files
    3. Auto-commit in child so it starts from a clean base
    4. Child does its work
    5. Rebase child onto updated parent (pulls interim changes from other children)
    6. Merge child branch back to parent (sequential, conflict-safe)
    7. Run merge-test gate between merges
    8. Clean up worktree + branch

    Composes with existing sdlc_worktree.py primitives + new state-capture layer.

    P0/P1 fixes from advisor review applied.
    Corner-case mitigations from DeepSeek analysis applied.
    """
    import concurrent.futures
    import fnmatch
    import os
    import shutil
    import subprocess
    import tempfile
    import uuid

    results = {}

    # Kimi M7: Helper for git commands that must succeed — prevents silent
    # failures (missing user.name, gpg sign errors, etc.) from leaving the
    # worktree in a dirty state and causing confusing downstream errors.
    def _git_or_raise(cmd, cwd, timeout=10, ignore_nothing=False):
        r = subprocess.run(cmd, cwd=cwd, capture_output=True, text=True, timeout=timeout)
        if r.returncode != 0:
            err = (r.stderr or "").lower()
            if ignore_nothing and ("nothing to commit" in err or "nothing added" in err):
                return r
            raise RuntimeError(f"git {' '.join(cmd)} failed in {cwd}: {r.stderr}")
        return r

    # DeepSeek E1: Empty repo detection — git worktree add fails without commits
    head_check = subprocess.run(
        ["git", "rev-parse", "--verify", "HEAD"],
        cwd=worktree, capture_output=True, text=True, timeout=10
    )
    if head_check.returncode != 0:
        raise RuntimeError("Cannot dispatch parallel tasks: repo has no commits yet")

    # DeepSeek E2: Detached HEAD detection — merge-back target is ambiguous
    branch_r = subprocess.run(
        ["git", "branch", "--show-current"],
        cwd=worktree, capture_output=True, text=True, timeout=10
    )
    if not branch_r.stdout.strip():
        raise RuntimeError("Cannot dispatch parallel tasks: orchestrator is in detached HEAD state")

    # DeepSeek E3: Mid-rebase/merge detection — git state is inconsistent
    # Kimi M3: Use `git rev-parse --git-path` instead of hardcoded `.git/` paths.
    # In linked worktrees, `.git` is a file (not a directory), and per-worktree
    # state is stored under `.git/worktrees/<name>/`. Hardcoded paths miss this.
    for state_name in ["rebase-merge", "rebase-apply", "MERGE_HEAD"]:
        path_r = subprocess.run(
            ["git", "rev-parse", "--git-path", state_name],
            cwd=worktree, capture_output=True, text=True, timeout=10
        )
        state_path = path_r.stdout.strip()
        if state_path and os.path.exists(os.path.join(worktree, state_path)):
            raise RuntimeError(f"Cannot dispatch parallel tasks: git is in mid-rebase/merge state ({state_name})")

    # ── Phase A: Capture orchestrator's uncommitted state ──────────────
    # Use git diff HEAD --binary (not git stash or git commit) so we don't
    # mutate the orchestrator's working state or pollute git history.
    run_id = uuid.uuid4().hex[:8]

    # A1: Capture tracked changes as a binary-safe patch
    patch_file = os.path.join(tempfile.gettempdir(), f"sdlc-patch-{run_id}.diff")
    with open(patch_file, 'wb') as f:
        r = subprocess.run(
            ["git", "diff", "HEAD", "--binary"],
            cwd=worktree, capture_output=True, timeout=30
        )
        f.write(r.stdout)

    # Kimi LOW #14: Cap patch size (DeepSeek C9/E6) — prevent memory exhaustion
    # from very large binary diffs on tmpfs. Fall back to commit-based capture.
    MAX_PATCH_BYTES = 50 * 1024 * 1024
    patch_size = os.path.getsize(patch_file)
    if patch_size > MAX_PATCH_BYTES:
        raise RuntimeError(
            f"Uncommitted patch is {patch_size} bytes (max {MAX_PATCH_BYTES}); "
            "fall back to commit-based state capture"
        )

    # Kimi M4: Read patch once into bytes to avoid leaking file handles.
    # Previously, `stdin=open(patch_file, 'rb')` was called per-child without
    # closing, leaking one fd per child until GC.
    with open(patch_file, 'rb') as f:
        patch_bytes = f.read()

    # A2: Capture untracked files (filtered — exclude .sdlc/, caches, .env)
    untracked_r = subprocess.run(
        ["git", "ls-files", "--others", "--exclude-standard"],
        cwd=worktree, capture_output=True, text=True, timeout=10
    )
    UNTRACKED_EXCLUDE = [
        '.sdlc/', '__pycache__/', '*.pyc', '.pytest_cache/',
        '.mypy_cache/', '.ruff_cache/', '*.egg-info/',
        'venv/', '.venv/', 'env/', 'node_modules/',
        '.env', '*.log', '.DS_Store',
    ]

    # Kimi M6: Fix exclude overmatch — `.sdlc/` should NOT match `.sdlc_backup/`
    def _is_excluded(path: str, pattern: str) -> bool:
        if pattern.endswith('/'):
            return path.startswith(pattern) or path.startswith(pattern.rstrip('/') + '/')
        return fnmatch.fnmatch(path, pattern)

    untracked_files = []
    for f in untracked_r.stdout.strip().split('\n'):
        if not f:
            continue
        excluded = any(_is_excluded(f, pat) for pat in UNTRACKED_EXCLUDE)
        if not excluded:
            untracked_files.append(f)

    # A3: Determine base_ref and capture HEAD hash for patch validation
    if base_ref is None:
        branch_r = subprocess.run(
            ["git", "branch", "--show-current"],
            cwd=worktree, capture_output=True, text=True, timeout=10
        )
        base_ref = branch_r.stdout.strip() or "HEAD"
    head_hash_r = subprocess.run(
        ["git", "rev-parse", "HEAD"],
        cwd=worktree, capture_output=True, text=True, timeout=10
    )
    head_hash = head_hash_r.stdout.strip()

    # ── Phase B: Create child worktrees + apply state ──────────────────
    from sdlc_worktree import create_worktree, remove_worktree
    task_worktrees = {}
    created_worktrees = []
    removed_worktrees = set()  # Kimi HIGH #2: Track removals to catch early-exit leaks

    try:
        for i, task in enumerate(tasks):
            task = dict(task)  # Kimi LOW #13: Shallow-copy to avoid mutating caller's dicts
            task["alias"] = None
            task["output_file"] = None
            tid = task.get("task_id", f"task_{i}")

            # B1: Create worktree from HEAD (same commit as patch base)
            # Kimi LOW #11: Removed dead `branch_name` variable — create_worktree()
            # generates its own branch name internally.
            try:
                wt_path = create_worktree(
                    message=f"parallel-{tid}",
                    base_ref=head_hash,  # Use commit hash, not branch name
                )
            except Exception as e:
                for cleanup_wt in created_worktrees:
                    try: remove_worktree(cleanup_wt)
                    except Exception: pass
                raise RuntimeError(f"Failed to create worktree for {tid}: {e}")

            task_worktrees[tid] = wt_path
            created_worktrees.append(wt_path)

            # B2: Assert child HEAD matches patch base
            child_head_r = subprocess.run(
                ["git", "rev-parse", "HEAD"],
                cwd=wt_path, capture_output=True, text=True, timeout=10
            )
            if child_head_r.stdout.strip() != head_hash:
                raise RuntimeError(
                    f"Child HEAD mismatch for {tid}: "
                    f"expected {head_hash}, got {child_head_r.stdout.strip()}"
                )

            # B3: Apply the patch in the child worktree
            # Kimi M4: Use pre-read patch_bytes via input= instead of stdin=open()
            if patch_bytes:
                apply_r = subprocess.run(
                    ["git", "apply", "--index"],
                    cwd=wt_path, input=patch_bytes,
                    capture_output=True, timeout=30
                )
                if apply_r.returncode != 0:
                    raise RuntimeError(
                        f"Patch apply failed for {tid}: {apply_r.stderr}"
                    )

            # B4: Copy filtered untracked files
            # Kimi LOW #12: Preserve symlinks instead of following them
            for f in untracked_files:
                src = os.path.join(worktree, f)
                dst = os.path.join(wt_path, f)
                if os.path.lexists(src):
                    os.makedirs(os.path.dirname(dst) or '.', exist_ok=True)
                    if os.path.islink(src):
                        link_target = os.readlink(src)
                        if os.path.lexists(dst):
                            os.remove(dst)
                        os.symlink(link_target, dst)
                    else:
                        shutil.copy2(src, dst)

            # B5: Auto-commit the applied state so child starts from clean base
            # Kimi M5: Use git add -f -A (force-add) so that inherited .gitignore
            # changes from the patch don't silently skip copied untracked files.
            _git_or_raise(["git", "add", "-f", "-A"], cwd=wt_path)
            _git_or_raise(
                ["git", "commit", "-m", f"checkpoint: inherit orchestrator state ({tid})",
                 "--allow-empty"],
                cwd=wt_path, ignore_nothing=True
            )

            task["cwd"] = wt_path

        # ── Phase C: Dispatch in parallel ──────────────────────────────
        with concurrent.futures.ThreadPoolExecutor(max_workers=max_workers) as pool:
            futures = {}
            for task in tasks:
                tid = task.pop("task_id", f"task_{len(futures)}")
                future = pool.submit(dispatch_single, **task)
                futures[future] = tid

            for future in concurrent.futures.as_completed(futures):
                tid = futures[future]
                try:
                    r = future.result(timeout=timeout)
                except Exception as e:
                    r = {"content": None, "error": str(e),
                          "elapsed": 0, "session_id": None}
                r["task_id"] = tid
                r["worktree"] = task_worktrees.get(tid)
                results[tid] = r
                if progress_callback:
                    _safe_callback(progress_callback, {
                        "event": "task_complete", "task_id": tid,
                        "success": r.get("content") is not None,
                        "elapsed": r.get("elapsed", 0),
                    })

    finally:
        # ── Phase D: Merge back (sequential, conflict-safe) ─────────────
        failed_tids = []
        for tid, result in results.items():
            if tid.startswith("__sdlc_"):  # Skip metadata keys
                continue
            if not isinstance(result, dict):
                continue
            wt_path = result.get("worktree")
            if not wt_path or not result.get("content"):
                failed_tids.append(tid)
                if wt_path:
                    try: remove_worktree(wt_path)
                    except Exception: pass
                    removed_worktrees.add(wt_path)
                continue

            # DeepSeek M5: Pre-merge — commit orchestrator's working state so
            # git merge doesn't fail with "local changes would be overwritten"
            _git_or_raise(["git", "add", "-A"], cwd=worktree)
            dirty_r = subprocess.run(
                ["git", "diff", "--cached", "--quiet"],
                cwd=worktree, capture_output=True, timeout=10
            )
            if dirty_r.returncode != 0:
                _git_or_raise(
                    ["git", "commit", "-m", "checkpoint: pre-merge orchestrator state"],
                    cwd=worktree, ignore_nothing=True
                )

            # D1: Auto-commit child's uncommitted work before rebase
            _git_or_raise(["git", "add", "-A"], cwd=wt_path)
            r = subprocess.run(
                ["git", "diff", "--cached", "--quiet"],
                cwd=wt_path, capture_output=True, timeout=10
            )
            if r.returncode != 0:  # Has staged changes
                _git_or_raise(
                    ["git", "commit", "-m", f"child: auto-commit uncommitted work ({tid})"],
                    cwd=wt_path, ignore_nothing=True
                )

            # D2: Rebase child onto updated parent (pulls interim changes)
            rebase_r = subprocess.run(
                ["git", "rebase", base_ref],
                cwd=wt_path, capture_output=True, text=True, timeout=60
            )
            if rebase_r.returncode != 0:
                # Check for .sdlc/ conflicts (auto-resolve, keep child's)
                conflict_r = subprocess.run(
                    ["git", "diff", "--name-only", "--diff-filter=U"],
                    cwd=wt_path, capture_output=True, text=True, timeout=10
                )
                conflicts = conflict_r.stdout.strip().split('\n') if conflict_r.stdout.strip() else []
                sdlc_only = all(f.startswith('.sdlc/') for f in conflicts) if conflicts else False
                if conflicts and sdlc_only:
                    for f in conflicts:
                        subprocess.run(["git", "checkout", "--theirs", f],
                                       cwd=wt_path, capture_output=True, timeout=5)
                        subprocess.run(["git", "add", f], cwd=wt_path,
                                       capture_output=True, timeout=5)
                    # Kimi M8: Check rebase --continue returncode — if it fails,
                    # abort + fail instead of falling through to merge while
                    # still mid-rebase.
                    continue_r = subprocess.run(
                        ["git", "rebase", "--continue"], cwd=wt_path,
                        capture_output=True, text=True, timeout=30
                    )
                    if continue_r.returncode != 0:
                        subprocess.run(["git", "rebase", "--abort"], cwd=wt_path,
                                       capture_output=True, timeout=10)
                        result["merge_success"] = False
                        result["merge_conflicts"] = continue_r.stderr or "rebase --continue failed"
                        failed_tids.append(tid)
                        try: remove_worktree(wt_path)
                        except Exception: pass
                        removed_worktrees.add(wt_path)
                        continue
                else:
                    subprocess.run(["git", "rebase", "--abort"], cwd=wt_path,
                                   capture_output=True, timeout=10)
                    result["merge_success"] = False
                    result["merge_conflicts"] = "; ".join(conflicts) if conflicts else str(rebase_r.stderr)
                    failed_tids.append(tid)
                    try: remove_worktree(wt_path)
                    except Exception: pass
                    removed_worktrees.add(wt_path)
                    continue

            # D3: Merge child branch back to parent
            child_branch = subprocess.run(
                ["git", "branch", "--show-current"],
                cwd=wt_path, capture_output=True, text=True, timeout=10
            ).stdout.strip()

            merge_r = subprocess.run(
                ["git", "merge", "--no-ff", child_branch,
                 "-m", f"Merge parallel/{tid}"],
                cwd=worktree, capture_output=True, text=True, timeout=30
            )
            if merge_r.returncode != 0:
                subprocess.run(["git", "merge", "--abort"], cwd=worktree,
                               capture_output=True, timeout=10)
                result["merge_success"] = False
                result["merge_conflicts"] = merge_r.stderr or merge_r.stdout
                failed_tids.append(tid)
            else:
                result["merge_success"] = True
                result["merge_conflicts"] = ""

                # Kimi M9: Delete child branch after successful merge —
                # remove_worktree() doesn't delete the branch, so they
                # accumulate and clutter `git branch`.
                subprocess.run(
                    ["git", "branch", "-d", child_branch],
                    cwd=worktree, capture_output=True, timeout=10
                )

                # DeepSeek §4.3: Merge-test gate — run tests after each child
                # merge to catch integration bugs early. If tests break after
                # merging this child, stop merging remaining children and enter
                # DEBUG state.
                # NOTE: This is a design-level placeholder. The actual test
                # command depends on the project (pytest, unittest, etc.)
                # and should be passed as a parameter or read from config.
                # test_result = run_tests_in_worktree(worktree)
                # if not test_result["passed"]:
                #     _emit(f"  ⚠️ merge of {tid} introduced test failures")
                #     result["merge_test_failed"] = True
                #     # Stop merging remaining children — enter DEBUG
                #     break

            # E: Clean up worktree
            try: remove_worktree(wt_path)
            except Exception: pass
            removed_worktrees.add(wt_path)

        # Kimi HIGH #2: Sweep created_worktrees for any that were never added
        # to results (e.g., exception after worktree creation but before
        # dispatch). Without this, they leak permanently.
        for wt_path in created_worktrees:
            if wt_path not in removed_worktrees:
                try: remove_worktree(wt_path)
                except Exception: pass
                removed_worktrees.add(wt_path)

        # Kimi M10: Use namespaced key to prevent collision with task_id="_merge_failures"
        if failed_tids:
            results["__sdlc_merge_failures__"] = failed_tids

        # Clean up temp patch file
        try: os.remove(patch_file)
        except Exception: pass

    return results
```

### 2.4 Integration into the State Machine

The PLAN phase is the right place to identify parallelizable tasks. The planner must
emit a structured task list with file paths (not free text) for parallelizable tasks:

**P1 fix per advisor review:** The heuristic "look for `## Task N` sections" approach
is too fragile for production. The planner prompt must explicitly request structured
output when parallelizable tasks exist, with file-independence validation across tasks.
If parsing fails or yields 1 task, fall back to sequential dispatch.

```python
# In PLAN state, after dispatch_single returns:
# Parse plan for structured task definitions
# The planner prompt instructs: "If the problem has independent modules,
# output a JSON array of tasks with id, prompt, and files[] fields.
# Each task's files[] MUST NOT overlap with other tasks' files[]."
plan_tasks = parse_plan_tasks(run.last_plan)
# → [{"id": "module_a", "prompt": "...", "files": ["module_a.py", "test_module_a.py"]},
#    {"id": "module_b", "prompt": "...", "files": ["module_b.py", "test_module_b.py"]},
#    ...]

# P1 fix: Validate file-set independence across tasks
if len(plan_tasks) > 1:
    all_files = [f for t in plan_tasks for f in t["files"]]
    if len(all_files) != len(set(all_files)):
        # File overlap detected — fall back to sequential
        _emit("  ⚠️ Parallel tasks have file overlap — using sequential dispatch")
        plan_tasks = []

if len(plan_tasks) > 1:
    # Parallel dispatch path
    tasks = []
    for pt in plan_tasks:
        tasks.append({
            "task_id": pt["id"],
            "model": MODEL_CODER_V6,
            "prompt": pt["prompt"],
            "context": "",
            "toolsets": "terminal,file",
            "max_turns": None,
            "timeout": TIMEOUT_DISPATCH_V6,
            # P2 fix: Derive provider from model, don't hardcode
            "provider": get_provider_for_model(MODEL_CODER_V6),
            "thinking": None,
            "role": "coder",
            "cwd": None,  # set by dispatch_parallel_isolated
        })
    results = dispatch_parallel_isolated(tasks, worktree, max_workers=3)
    # Process results: check which succeeded, which failed
    successful = [r for r in results.values() if isinstance(r, dict) and r.get("content")]
    failed = [r for r in results.values() if isinstance(r, dict) and not r.get("content")]
    merge_failures = results.get("_merge_failures", [])

    if failed:
        _emit(f"  ⚠️ {len(failed)}/{len(tasks)} parallel tasks failed")
    if merge_failures:
        _emit(f"  ⚠️ {len(merge_failures)} merge failures: {merge_failures}")
        # P1 fix: Feed conflicting task IDs back to next PLAN iteration
        # so the planner keeps those modules separate
        run.parallel_merge_failures = merge_failures

    # All tasks done — proceed to LINT_FIX (merge already happened)
    state = SDLCState.LINT_FIX
else:
    # Sequential dispatch path (current behavior)
    # Also used when: plan parsing fails, file overlap detected, or 1 task
    state = SDLCState.IMPLEMENT
```

### 2.5 Completion on Child Exit — Barrier Semantics

In Option A, the orchestrator uses a **barrier**: `concurrent.futures.as_completed()`
processes results as they arrive, but the orchestrator doesn't transition state until
**all** children have exited. This is the simplest correct semantics:

```
Orchestrator dispatches 3 children
    │
    ├── child_1 exits (45s) → result collected, orchestrator waits
    ├── child_2 exits (67s) → result collected, orchestrator waits
    └── child_3 exits (103s) → result collected → ALL DONE → merge → LINT_FIX
```

**Why barrier (not streaming):**
- LINT_FIX needs all files present to run ruff/mypy across the full codebase
- RUN_TESTS needs all test files to collect the complete test suite
- Partial testing would give misleading pass/fail counts

**Future: streaming for non-file tasks.** If we add independent review tasks (e.g., 3
verifiers reviewing different acceptance criteria), those could be processed as they
complete without a barrier — each result is independent.

### 2.6 Constraints and Limitations of Option A

| Constraint | Impact | Mitigation |
|-----------|--------|------------|
| `thinking=None` required for all parallel tasks | Can't use different reasoning levels per task | Use same thinking for all, or serialize. Future: `--reasoning-effort` CLI flag |
| Shared Ollama endpoint | N concurrent requests to same local model | **RESOLVED (2026-06-29):** Benchmark confirmed 3.25x speedup — Ollama handles concurrent requests without serializing. 3 sequential tasks = 42s, 3 parallel = 12.9s. No contention observed. |
| `git worktree add` overhead | ~0.5s per task, creates temp directories | Cleanup in `finally` block; acceptable for 2-5 tasks |
| Merge conflicts | Two tasks touching same file → conflict | Plan must identify truly independent tasks (different files) |
| No child→orchestrator messaging | Child can't ask questions mid-flight | See Phase 2 below |
| No cancellation | Can't kill a running subprocess mid-flight | `future.cancel()` only cancels un-started tasks; running tasks complete. Future: `subprocess.Popen` + `process.kill()` |
| Wall-clock budget | Long-running parallel task blocks barrier | Per-task timeout = TIMEOUT_DISPATCH_V6 (3600s); orchestrator wall-clock catches at next loop iteration |

## 3. Phase 2 — API-Based Child↔Orchestrator Communication

### 3.1 The Core Problem

Today, the child process is:
```
hermes chat -q <prompt> -m <model> --provider <provider> -Q --yolo --pass-session-id
```

This is:
- **Headless**: `-Q` = quiet mode (stdout only, no interactive prompt)
- **Autonomous**: `--yolo` = no approval prompts (all tool calls auto-approved)
- **One-shot**: returns one text blob when the model stops calling tools or hits max_turns
- **No bidirectional communication**: the orchestrator sends a prompt, the child sends back text. No mid-flight conversation.

The child **cannot ask the orchestrator a question** because:
1. It has no `clarify` tool (leaf agents don't get it)
2. Even if it did, `clarify` would prompt the user's terminal — but there is no user terminal (it's `-Q --yolo`)
3. The orchestrator is blocked on `subprocess.run()` — it can't respond to anything the child sends

### 3.2 Three Integration Pathways

#### Pathway 1: `--resume` for Multi-Turn Child Sessions (Near-Term)

The child gets a `session_id` on exit. The orchestrator can resume that session:

```python
# First dispatch — child writes code, encounters ambiguity
result = dispatch_single(
    model=MODEL_CODER_V6,
    prompt="Implement module_a.py and test_module_a.py...",
    toolsets="terminal,file",
    ...
)
session_id = result["session_id"]

# Orchestrator inspects result — detects ambiguity in output
if "I'm not sure if" in result["content"] or "unclear" in result["content"]:
    # Resume the child session with clarification
    result2 = dispatch_single(
        model=MODEL_CODER_V6,
        prompt=f"The answer to your question is: the config should be a dict, not a list. "
               f"Continue implementing.",
        toolsets="terminal,file",
        resume_session=session_id,  # ← child remembers its previous context!
        ...
    )
```

**How it works:**
- First dispatch: child runs, hits ambiguity, makes a guess (or writes partial code)
- `dispatch_single` captures `session_id` from `--pass-session-id` output
- Orchestrator inspects child's output for uncertainty markers
- Second dispatch: `--resume <session_id>` — child picks up where it left off, with
  the orchestrator's answer injected as a new user message
- Child continues with full context of its prior turns

**Limitations:**
- Still one-shot per dispatch — the child can't ask mid-flight, only on exit
- Orchestrator must detect the question in the child's output (pattern matching)
- Requires the orchestrator to "understand" what the child is uncertain about
- Session resume has context overhead — the child re-loads full conversation history

**When to use:** Low-frequency clarification. Most dispatches won't need it. The
orchestrator checks for uncertainty markers and only resumes when needed.

#### Pathway 2: Structured Output Protocol (Medium-Term)

The child returns structured JSON instead of free text, including a "questions" field:

```json
{
  "status": "blocked",
  "questions": [
    "PROJECT.md says 'validate input' but doesn't specify which types. Should I validate strings, numbers, or both?"
  ],
  "partial_work": {
    "files_written": ["module_a.py"],
    "files_pending": ["test_module_a.py"]
  }
}
```

The orchestrator parses this, answers the questions, and re-dispatches with the answers:

```python
result = dispatch_single(model=MODEL_CODER_V6, prompt="...", toolsets="terminal,file")

# Parse structured output
child_output = parse_child_output(result["content"])
if child_output.get("status") == "blocked" and child_output.get("questions"):
    # Orchestrator answers questions (or proxies to user)
    answers = {}
    for q in child_output["questions"]:
        # Try to answer from PROJECT.md context
        answer = find_answer_in_project_md(q, run.project_md)
        if answer:
            answers[q] = answer
        else:
            # Can't answer — route to HUMAN_REVIEW
            run.pending_questions = child_output["questions"]
            state = SDLCState.HUMAN_REVIEW
            break
    else:
        # All questions answered — resume child session with answers
        answer_text = format_answers(answers)
        result2 = dispatch_single(
            model=MODEL_CODER_V6,
            prompt=f"Here are the answers to your questions:\n{answer_text}\nContinue.",
            resume_session=result["session_id"],
            ...
        )
```

**How it works:**
- The child's prompt instructs it to output JSON with a `status` field and `questions` array
- The orchestrator parses the JSON, routes questions to PROJECT.md lookup or HUMAN_REVIEW
- On resume, the child gets the answers and continues

**Limitations:**
- Requires prompt engineering to get reliable JSON output from local models
- Still one-shot per dispatch — the child asks on exit, not mid-flight
- Adds parsing complexity
- Local models may not produce valid JSON reliably

**When to use:** When you want structured communication without changing the dispatch
primitive. Works with any model that can follow output format instructions.

#### Pathway 3: Bidirectional IPC via Hermes API Server (Long-Term)

Hermes has an API server mode (`api-server` source in sessions docs). Instead of
`subprocess.run(hermes chat -q ...)`, the orchestrator would make HTTP API calls to
a running Hermes agent instance:

```python
# Start a Hermes API server for the child
child_server = start_hermes_api_server(model=MODEL_CODER_V6, cwd=worktree)

# The child agent runs as a server, not a subprocess
# The orchestrator sends prompts via HTTP:
response = requests.post(f"http://localhost:{child_server.port}/chat", json={
    "message": "Implement module_a.py...",
    "toolsets": "terminal,file",
})

# The child can call back to the orchestrator via a webhook/tool:
# The child's toolset includes a custom "ask_orchestrator" tool:
response2 = requests.post(f"http://localhost:{child_server.port}/chat", json={
    "message": "Continue implementing with these clarifications: ...",
})
```

And the child side — the child agent gets a custom tool:

```python
# Child agent has this tool available:
def ask_orchestrator(question: str) -> str:
    """Ask the orchestrator a question and get an answer."""
    response = requests.post("http://localhost:{ORCHESTRATOR_PORT}/child-question", json={
        "question": question,
        "session_id": CHILD_SESSION_ID,
    })
    return response.json()["answer"]
```

**How it works:**
1. Orchestrator starts a Hermes API server for the child (or uses a shared server with per-session routing)
2. Orchestrator sends initial prompt via HTTP POST
3. Child agent starts processing — if it hits ambiguity, it calls `ask_orchestrator(question)`
4. The `ask_orchestrator` tool makes an HTTP callback to the orchestrator
5. Orchestrator receives the question, answers it (from PROJECT.md, or proxies to user)
6. Child agent gets the answer and continues
7. Child signals completion via HTTP response

**This is true bidirectional communication — the child can ask questions mid-flight.**

**Limitations:**
- Requires Hermes API server mode (exists but needs investigation for headless/agent use)
- Requires custom tool registration (`ask_orchestrator`) — Hermes supports custom tools
- Orchestrator needs an HTTP server running to receive callbacks (async or threaded)
- Significantly more complex than subprocess.run()
- Session management across HTTP boundaries

**When to use:** When children frequently need clarification and one-shot dispatch
isn't enough. This is the "conversational child" model — the child and orchestrator
have a back-and-forth dialogue.

### 3.3 How Completion Works in Each Pathway

| Pathway | Completion Signal | What Orchestrator Does |
|---------|-------------------|----------------------|
| Current (subprocess.run) | Process exits → `subprocess.run()` returns | Parses stdout, processes content |
| Pathway 1 (--resume) | Process exits → orchestrator checks for questions → may resume | Detects uncertainty, answers, re-dispatches with --resume |
| Pathway 2 (structured output) | Process exits → orchestrator parses JSON → may answer questions and resume | Parses JSON, routes questions, resumes if answerable |
| Pathway 3 (API server) | HTTP response received (or timeout) | Sends prompt, receives callback for questions, sends answers, waits for final response |

### 3.4 How the Orchestrator Proxies to the User

In all three pathways, the orchestrator is the user's only interface:

```
Child: "Should I validate strings or numbers?"
    ↓
Orchestrator: (checks PROJECT.md)
    ├── Found answer in PROJECT.md → sends back automatically (no user involved)
    ├── No answer in PROJECT.md → routes to HUMAN_REVIEW
    │     ↓
    │   User sees: "Child asks: Should I validate strings or numbers?"
    │   User responds: "Both — validate any input type"
    │     ↓
    └── Orchestrator sends user's answer back to child via --resume / API / structured output
```

The orchestrator is a **proxy layer**: it intercepts child questions, tries to answer
from context (PROJECT.md, learnings, prior iterations), and only escalates to the
user when it can't answer. This minimizes user interruption while giving children
the ability to get unstuck.

### 3.5 Recommended Migration Path

```
Today:     subprocess.run() — one-shot, blocking, no messaging
                ↓
Phase 1:   dispatch_parallel() — concurrent, barrier, no messaging
           (ThreadPoolExecutor + git worktree isolation)
                ↓
Phase 1.5: --resume + uncertainty detection
           (child exits → orchestrator checks for questions → resumes with answers)
           Low-effort, works with existing infrastructure
                ↓
Phase 2:   Structured output protocol
           (child returns JSON with questions field → orchestrator parses → resumes)
           Medium-effort, requires prompt engineering + JSON parsing
                ↓
Phase 3:   API server + custom ask_orchestrator tool
           (child calls back to orchestrator mid-flight via HTTP)
           High-effort, true bidirectional communication
           Requires Hermes API server investigation + custom tool registration
```

## 5. Key Design Decisions

1. **Barrier semantics for Phase 1** — wait for all children before transitioning state.
   This is simplest and correct for file-producing tasks. Streaming completion is a
   future optimization for non-file tasks (review, verification).

2. **`git worktree add` for isolation** — not branches, not temp directories. Git worktrees
   are the correct primitive: separate working directory, shared .git, branch-per-task,
   merge after completion. ~0.5s overhead per task, no race conditions.

3. **`thinking=None` for all parallel tasks** — the global config race is a hard constraint
   until Hermes adds a `--reasoning-effort` CLI flag. All parallel tasks inherit the Hermes
   default. If different thinking levels are needed, fall back to sequential.

4. **--resume as the near-term messaging bridge** — the child exits with a session_id.
   The orchestrator can detect uncertainty in the output and resume the session with
   answers. This works with existing infrastructure (no API server, no custom tools)
   and gives us 80% of the value with 20% of the effort.

5. **Orchestrator as proxy, not passthrough** — the orchestrator tries to answer child
   questions from PROJECT.md/learnings/context before escalating to the user. This
   minimizes user interruption and keeps the loop autonomous.

6. **Completion = child exit (Phase 1) → HTTP response (Phase 3)** — in Phase 1, the
   subprocess exiting IS the completion signal. In Phase 3, the HTTP response IS the
   completion signal (which may include mid-flight question callbacks). The orchestrator
   always owns the decision to transition state.

## 6. Open Questions

1. **~~Ollama concurrency~~** — ~~can the local Ollama proxy handle 3 concurrent requests
   to the same model?~~ **RESOLVED (2026-06-29):** Benchmark confirmed 3.25x speedup.
   Ollama handles concurrent requests without serializing. Phase 1 speedup is real.

2. **Merge conflict frequency** — how often will independent plan tasks touch the same
   files? If the planner outputs truly independent tasks (different modules, different
   test files), conflicts should be rare. But shared imports, __init__.py, and config
   files are likely conflict points.

3. **Hermes API server for headless use** — does `hermes api-server` (or equivalent) support
   running a headless agent with tool access? Or is it just a chat completion endpoint?
   Needs investigation of Hermes source/docs.

4. **Custom tool registration** — can we register an `ask_orchestrator` tool in a child
   agent's toolset? Hermes supports custom tools via plugins — needs to be confirmed
   for subprocess/API-server mode.

5. **Wall-clock budget with parallel dispatch** — the top-of-loop check still works
   (checks elapsed before each state transition). But during a parallel dispatch, all
   children run simultaneously and the orchestrator is blocked on `as_completed()`.
   The per-task timeout (3600s) is the only protection. Should we cap per-task timeout
   to remaining wall-clock? (User previously rejected this — but that was for sequential
   dispatch. With parallel dispatch, the barrier means one slow task blocks all others.)
