## Lessons Learned from Live E2E Testing

### What Actually Broke

During 5 live E2E tests (palindrome, fizzbuzz, fibonacci, factorial, debug_iteration),
we hit **3 distinct failure modes** that the pipeline didn't handle:

1. **API errors returned as content** — HTTP 429 (Ollama monthly rate limit) was
   returned in `dispatch_result['content']` as a string, not in the `error` field.
   The E2E test tried to execute `"API call failed after 3 retries: HTTP 429..."`
   as Python code → `SyntaxError`. We patched this in the *test*, not the pipeline.

2. **Triage classification was non-deterministic** — The palindrome idea was
   classified as `debug_code` on one run and `build_code` on another. Tests had
   to be made category-flexible. The pipeline has no notion of "close enough"
   categories or fallback routing.

3. **Model produced code-review diff format, not standalone code** — GLM-5.2
   (the current model) acting as a code reviewer returned a unified diff
   (`┊ review diff a//path → b//path`) instead of a self-contained script.
   The code extractor couldn't parse this. We fixed the *prompt*, not the pipeline.

### What Was Known But Never Fixed

4. **`role` passthrough gap** — `routing.py` assigns `role='debugger'` for
   `debug_code` category, but `pipeline.py` never passes `role` to
   `dispatch_single()`. The role is computed and discarded. (Flagged by Kimi
   review, never implemented.)

### P9: SDLC Expansion — Cascading Debugger + Test-First Pipeline — ✅ DONE

Expanded the SDLC pipeline from single-dispatch to multi-phase test-first with
a cascading debugger. Architecture:

```
build_code → Phase 1: plan()           — GLM inspects existing, plans (KISS/YAGNI)
             Phase 2: design_test_suites() — DeepSeek designs 3 suites (unit/integration/e2e)
             Phase 3: implement()       — Qwen-coder writes code given plan + tests
             Phase 4: run_test_suites() — Execute pytest against generated code
             Phase 5: debug_cascade()   — If tests fail: qwen-coder → kimi

debug_code → debug_cascade()            — qwen-coder (attempt 1) → kimi (attempt 2)
```

Key design decisions: cascade not parallel (cheap model first, escalate on failure),
reuse directive (inspect existing before designing new), test-first not code-first,
`no_verify` flag for quick prototyping.

### Bugs Found & Fixed During P12

**Bug 1: `toolsets='web'` + `max_turns=5` causes models to attempt tool calls** — 🔧 FIXED

Models with tool access (toolsets='web', max_turns=5) attempted `execute_code`
tool calls instead of generating output text. Affected 4 functions:

| Function | File:line | Fix |
|----------|----------|-----|
| `implement()` | sdlc.py:307 | `toolsets=''`, `max_turns=1` |
| `tech_docs()` | sdlc.py:706 | `toolsets=''`, `max_turns=1` |
| `simplify_code()` | sdlc.py:779 | `toolsets=''`, `max_turns=1` |
| `council_review()._dispatch_seat()` | sdlc.py:878 | `toolsets=''`, `max_turns=1` |

**Root cause:** The pipeline passes `toolsets` from routing into SDLC phase
functions. When routing assigns `toolsets='web'` (for build_code), the generation
phases inherit it — but these phases should output code/text directly, not call
tools. The prompt already says "Do not use file tools" but models with tool
access ignore that and call tools anyway.

**Lesson:** Generation phases (implement, tech_docs, simplify, council) must
always use `toolsets=''` and `max_turns=1`. Only routing/inspection phases
(plan, design_test_suites) may use tools.

**Bug 2: `extract_python_code()` too strict — fails on valid code blocks** — 🔧 FIXED

The extraction function's Strategy 2 (generic ``` blocks) required Python
keywords (`def`, `import`, `class`, `print(`) to return code. If a model outputs
a code block without these keywords (e.g. a script starting with `if __name__`),
extraction returned None → pipeline silently "succeeded" with no code.

**Fix:** Added `return` and `if __` to the keyword list (Strategy 2), and added
a fallback that returns the block even without keywords. Also added a pipeline
guard: if `extracted_code` is None but `code_result['content']` has content,
try lenient extraction (any ``` block), then raw content if >50 chars with
Python keywords.

**Bug 3: Pipeline silently succeeds with `extracted_code=None`** — 🔧 FIXED

When `extract_python_code()` returned None, the pipeline continued to
`run_test_suites` with None code, or reported success with no code. Now the
pipeline returns `pipeline_status='implement_failed'` if no code can be
extracted after all fallback strategies.

**Bug 4: `extract_python_code()` takes first block, not largest** — 🔧 FIXED

Strategy 1 used `re.search` (first match only). Models may emit multiple
```python blocks (plan in one, code in another). Changed to `re.findall` +
return largest block.

### P14-H: Linting Phase — Code Lint + Auto-Fix Before Tests — ✅ DONE

**Problem:** Generated code AND generated tests often have style issues (trailing
whitespace, mixed tabs/spaces, unused imports) that could be caught and auto-fixed
before running tests. This reduces test failures caused by trivial formatting issues.

**Fix:** Added `lint_code()` and `lint_test_suites()` functions (sdlc.py) and two
new phases in the pipeline:

```
Pipeline flow:
  Phase 1: plan
  Phase 2: design_test_suites
  Phase 3: implement
  Phase 3.5: lint_code — syntax check + aggressive auto-fix ← NEW
  Phase 3.6: lint_test_suites — lint + auto-fix test code ← NEW
  Phase 4: run_test_suites (uses fixed code + fixed tests)
  Phase 5: debug_cascade (if tests fail)
  ...
```

**lint_code() aggressively does:**
1. Syntax check via `ast.parse()` — catches syntax errors before tests run
2. `py_compile` check — catches compilation errors
3. **Aggressive auto-fix** — runs multiple tools in sequence:
   - `ruff check --fix --unsafe-fixes` (lint fixes: unused imports, undefined names)
   - `ruff format` (formatting: line length, spacing, quotes)
   - `autopep8 --aggressive --aggressive` (fallback if ruff unavailable)
4. Re-verify fixed code with `ast.parse()` — revert if auto-fix broke syntax
5. Heuristic checks — trailing whitespace, mixed tabs/spaces (always run)

**lint_test_suites() does:**
- Lints each test suite individually with `lint_code(auto_fix=True)`
- Returns fixed_suites dict with auto-fixed test code
- Reports `any_failed` if any suite has syntax errors (doesn't block pipeline —
  pytest will fail and cascade handles it)

**Pipeline behavior:**
- Auto-fixed code replaces extracted_code for all subsequent phases
- Auto-fixed test suites replace original test_suites
- Syntax errors in code → `pipeline_status='lint_failed'`
- Syntax errors in tests → warning printed, tests will fail → cascade handles
- `lint_result` and `test_lint_result` in pipeline return dict

**Tests:** 16 new tests in `TestLintCode` class covering:
- Valid code passes, syntax errors detected, syntax errors not auto-fixed
- Trailing whitespace detected, mixed tabs/spaces detected
- Return dict structure, pipeline includes lint_result
- Pipeline returns lint_failed on syntax-broken code
- Aggressive auto-fix runs ruff check + ruff format + autopep8
- Auto-fix verifies syntax after fixing (reverts if broken)
- lint_test_suites returns correct structure
- lint_test_suites detects syntax errors
- lint_test_suites handles empty/None suites
- Pipeline includes test_lint_result
